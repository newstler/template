# Plan 08: Safe Delete (Trashable + DeletionAudit)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add restorable, auditable "safe delete" to the business-valuable records in the template (User, Team, Membership, Conversation, ConversationMessage, ConversationParticipant, Chat, Message), while keeping hard-delete for ephemeral data (Article, notifications, language join rows). A deletion captures a frozen display snapshot so the other party in a B2B relationship still sees a meaningful "Deactivated User" placeholder instead of a dangling FK. A `PurgeExpiredDeletionsJob` hard-deletes records whose retention window has lapsed. Madmin gets a "Deletions" dashboard resource for audit and manual restore/purge.

**Architecture:** State-as-record via a polymorphic `deletions` table, exposed through a single `Trashable` concern that models `include` and whose `trashable_snapshot_fields` class method declares the fields to freeze at deletion time. `has_one :deletion, as: :trashable, dependent: :destroy`. Scopes: `kept` (default) / `trashed` / `with_trashed`. Instance methods: `trash(by:, reason:, source:)`, `restore`, `trashed?`, `display_snapshot`. `belongs_to` relationships stay intact — a trashed parent is still loadable, it just answers `trashed?` and returns a frozen snapshot via `display_name` / `display_avatar_url`. A `TrashableCascade` option on the concern declares which associations cascade-trash (`trashable_cascade :memberships, :conversation_participants`) — distinct from `dependent: :destroy`. A `PurgeExpiredDeletionsJob` runs daily via Solid Queue recurring, hard-deletes records whose `deletion.expire_at` has passed, and keeps the `deletions` row itself as the audit trail. Retention windows are per-model via `Setting.*_retention_days` keys, defaulting to 30. Controller actions: `DELETE /profile` (self-trash), `DELETE /t/:slug` (team-trash by owner), `POST /profile/restore` (self-restore during window). Madmin ships `DeletionResource` + restore/purge member actions. MCP gets `trash_user`, `restore_user`, `trash_team`, `restore_team` tools.

**Tech Stack:** Rails 8, Ruby 4.0.x, SQLite + Solid Stack (Cache, Queue), UUIDv7 PKs via `uuid7()`, Hotwire (Turbo + Stimulus), Tailwind v4 OKLCH, Madmin, fast-mcp.

## Why this design (and not a gem)

**Picked:** custom `Trashable` concern + polymorphic `Deletion` state-as-record. Runner-up: `discard` gem.

One-paragraph comparison the plan is based on:

- **`paranoia` (`acts_as_paranoid`)** — callback-heavy; hijacks `destroy` to mean "soft delete" and rewrites default scopes. Famously fights with `dependent: :destroy` cascades, Rails upgrades, and any code that calls `record.destroy`. Rejected.
- **`discard`** — light, tag-based, single `discarded_at` column per model, maintained by Shopify. Would work, but requires per-model columns (conflicts with our "state as record" rule from `.claude/rules/ruby.md`), and has no opinion on snapshots, audit, cascading, or retention — we'd bolt those on anyway. Close second choice.
- **Event-sourcing / append-only log** — deletion as an event on a separate audit table. Overkill for our needs; we want "is it gone" + "who did it" + "can we restore," not full history.
- **Custom `Deletion` state-as-record (picked)** — one polymorphic table, one concern, rhymes exactly with the `Closeable` / `Closure` example the repo's own style guide uses as the canonical pattern (see `.claude/rules/ruby.md` lines 27–32). Frozen snapshot fits naturally as a JSON column on the `deletions` row. Restore is `deletion.destroy`. Purge is `record.destroy` after the window elapses. Zero gem dependency, zero monkey-patching of `ActiveRecord::Base#destroy`. Fits the "build it yourself for state and business logic" principle.

**Key architectural decisions (A–J from the spec):**

- **A. Pattern:** `Trashable` concern + polymorphic `Deletion` model. Explicit cascade via `trashable_cascade :assoc, ...` — we do NOT overload `dependent: :destroy`. Existing `dependent: :destroy` declarations stay where they are (for hard-delete cases like Language/TeamLanguage and for the purge path), and the concern adds its own cascade layer on top.
- **B. Scope:** Trashable = User, Team, Membership, Conversation, ConversationMessage, ConversationParticipant, Chat, Message. Hard-delete (no change) = Article, Language, TeamLanguage, Noticed::Notification, Noticed::Event, Setting, Admin, ProviderCredential. Justified model-by-model in Task 2.
- **C. Counterparty visibility:** `Deletion` row stores a `display_snapshot` JSON column frozen at trash time (`name`, `email`, `avatar_url`, `role`, `label` — configured per-model via `trashable_snapshot_fields`). The concern provides `display_name`, `display_email`, `display_avatar_url` helpers that return the snapshot when the record is trashed, or the live field when it isn't. Views use these helpers, not raw attributes. A shared `_deactivated_badge.html.erb` partial renders the strike-through treatment.
- **D. Restore:** two restore paths. (1) Self-service within the retention window for User and Team (visit the magic-link flow with a `restore=true` param, or call `POST /profile/restore` while logged in during a grace period). (2) Admin-only via Madmin for anything else. Restore un-deletes the root record and cascades-un-trash dependents *that were trashed as part of the same cascade event* (same `cascade_group_id` on the deletion row). Dependents trashed independently stay trashed. Dependents that have since been *hard*-purged (because *their* separate retention window elapsed) cannot be restored — the user sees a warning and the root is restored alone.
- **E. Purge:** `PurgeExpiredDeletionsJob` runs daily at 3am UTC via `config/recurring.yml`. Finds every `Deletion` where `expire_at < now`, calls `record.destroy` (the actual hard delete), then updates the deletion row with `purged_at: now` and keeps it forever as audit. The deletion row is orphaned at that point (its polymorphic target is gone), but still queryable by `trashable_type` + historical `trashable_id`. Retention windows come from `Setting` keys: `Setting.user_retention_days` (default 30), `Setting.team_retention_days` (default 30), `Setting.conversation_retention_days` (default 90), `Setting.message_retention_days` (default 90), `Setting.chat_retention_days` (default 30), `Setting.default_retention_days` (default 30 — fallback).
- **F. Audit:** the `Deletion` row itself IS the audit trail. Fields: `trashable_type`, `trashable_id`, `deleted_by_type`, `deleted_by_id` (polymorphic so both User and Admin work), `reason` (text, user-supplied), `source` (enum-like string: `ui`, `mcp`, `madmin`, `cascade`, `job`), `display_snapshot` (json — frozen fields), `cascade_group_id` (string, UUIDv7 — groups related cascaded deletions), `expire_at` (datetime — when the purge job will hard-delete), `purged_at` (datetime, nullable — set when the actual hard delete happens), `restored_at` (datetime, nullable — set on restore, then row is destroyed, so this is mainly for transient debugging / callbacks). Queryable from `/madmin/deletions`.
- **G. Concern vs per-model:** single `Trashable` concern. Every model that opts in includes it and declares `trashable_snapshot_fields` + optional `trashable_cascade`. No per-model concern files.
- **H. Association behavior:** `belongs_to` on a trashed parent *still loads* (SQL JOINs and `.find` work exactly as before — we do not rewrite `default_scope`). Use the `.kept` scope explicitly in any query that must hide trashed rows (controllers, index views). Use `.with_trashed` (the base table, no filter) in Madmin. Use the `display_*` helpers in any view that renders a parent-side association so the UI shows "Deactivated User" cleanly instead of dangling data. `has_many` relationships get a `.kept` scope on the through-association where relevant (e.g. `team.users.kept` vs `team.users.with_trashed`). `dependent: :destroy` declarations STAY — they describe what happens at the *final* hard-delete (purge) step, not at soft-delete time.
- **I. MCP surface:** `trash_user` (self-trash, Team + User auth), `restore_user` (self-restore within window), `trash_team` (team owner only), `restore_team` (team owner only). No admin-only MCP tools — admin restore/purge remain Madmin-only per the repo's "admin stays in Madmin" rule. No `trash_conversation` / `trash_chat` MCP tools in this plan — the existing `delete_chat` tool is repointed to `.trash` under the hood, so the MCP surface stays stable from the agent's perspective. Deferred: per-message trash as an MCP tool (can be added later if a consuming app needs it).
- **J. UI:** `DELETE /profile` (already absent — added in Task 11), `DELETE /t/:slug` (team self-destruct, owner-only), `POST /profile/restore` (within window), `POST /t/:slug/restore`. Team settings page gets a red-bordered "Danger zone" section. Profile edit page gets the same. A deleted-account banner partial is injected into any Turbo Frame that renders a trashed user's profile in someone else's view.

**What this plan does NOT cover** (deferred):
- Soft-delete of Noticed events and notifications (they're cheap and ephemeral; hard-delete is correct).
- Soft-delete of file attachments via Active Storage — attachments ride the parent's lifecycle. When a trashed parent is purged, `dependent: :purge_later` on `has_many_attached` handles blob cleanup.
- Legal-hold mechanism (freeze a record past its retention window for subpoena compliance). Add later via `deletion.legal_hold_until` column if a consuming app needs it.
- Hard-delete confirmation workflows (e.g. type-to-confirm in UI). A single-click `DELETE` with a flash-message confirmation is sufficient for this plan.

class Deletion < ApplicationRecord
  ALLOWED_SOURCES = %w[ui mcp madmin cascade job].freeze

  belongs_to :trashable, polymorphic: true, optional: true  # optional so purged rows still load
  belongs_to :deleted_by, polymorphic: true, optional: true

  validates :source, inclusion: { in: ALLOWED_SOURCES }
  validates :expire_at, presence: true

  scope :expired, -> { where("expire_at < ?", Time.current).where(purged_at: nil) }
  scope :purged, -> { where.not(purged_at: nil) }
  scope :live,   -> { where(purged_at: nil) }

  def expired? = expire_at < Time.current && purged_at.nil?
  def purged?  = purged_at.present?
end
```

```ruby
module Trashable
  extend ActiveSupport::Concern

  included do
    has_one :deletion, as: :trashable, dependent: :destroy
    scope :kept,         -> { where.missing(:deletion) }
    scope :trashed,      -> { joins(:deletion) }
    scope :with_trashed, -> { all }

    class_attribute :trashable_snapshot_fields, instance_writer: false, default: [:id]
    class_attribute :trashable_cascade_associations, instance_writer: false, default: []
  end

  class_methods do
    # DSL: declare which fields to freeze into the snapshot.
    #   trashable_snapshot :name, :email, :avatar_url
    def trashable_snapshot(*fields)
      self.trashable_snapshot_fields = (Array(trashable_snapshot_fields) + fields).uniq
    end

    # DSL: declare which associations to cascade-trash.
    #   trashable_cascade :memberships, :conversation_participants
    def trashable_cascade(*assocs)
      self.trashable_cascade_associations = (Array(trashable_cascade_associations) + assocs).uniq
    end

    # Retention window in days for this model class.
    # Reads Setting.<underscored_name>_retention_days, falls back to default_retention_days.
    def trashable_retention_days
      key = :"#{model_name.element}_retention_days"
      Setting.get(key).presence || Setting.get(:default_retention_days).presence || 30
    end
  end

  def trashed? = deletion.present?

  def trash(by:, reason: nil, source: "ui", cascade_group_id: nil)
    return deletion if trashed?

    group = cascade_group_id || SecureRandom.uuid
    expire_at = self.class.trashable_retention_days.to_i.days.from_now

    transaction do
      create_deletion!(
        deleted_by: by,
        reason: reason,
        source: source.to_s,
        display_snapshot: build_snapshot,
        cascade_group_id: group,
        expire_at: expire_at,
      )

      trashable_cascade_associations.each do |assoc_name|
        Array(public_send(assoc_name)).each do |child|
          next if !child.is_a?(ActiveRecord::Base) || !child.class.include?(Trashable)
          child.trash(by: by, reason: "cascade:#{self.class.name}", source: "cascade", cascade_group_id: group)
        end
      end
    end

    deletion
  end

  def restore
    return false unless trashed?

    transaction do
      group = deletion.cascade_group_id
      deletion.destroy!

      # Un-trash any siblings from the same cascade group that still exist.
      if group.present?
        Deletion.live.where(cascade_group_id: group).find_each do |d|
          d.trashable&.restore
        end
      end
    end

    reload
    true
  end

  def display_snapshot
    deletion&.display_snapshot || {}
  end

  def display_name
    trashed? ? display_snapshot["name"].presence || I18n.t("trashes.deactivated") : public_send_if_exists(:name)
  end

  def display_email
    trashed? ? display_snapshot["email"] : public_send_if_exists(:email)
  end

  def display_avatar_url
    trashed? ? display_snapshot["avatar_url"] : nil
  end

  private

  def build_snapshot
    trashable_snapshot_fields.each_with_object({}) do |field, acc|
      acc[field.to_s] = public_send_if_exists(field)
    end.compact
  end

  def public_send_if_exists(method_name)
    respond_to?(method_name) ? public_send(method_name) : nil
  end
end
```

class PurgeExpiredDeletionsJob < ApplicationJob
  queue_as :low_priority

  def perform
    Deletion.expired.includes(:trashable).find_each do |deletion|
      target = deletion.trashable
      deletion.update_columns(purged_at: Time.current)

      # Actual hard-delete. dependent: :destroy on the various
      # associations takes it from here. If a target has already been
      # destroyed (rare race), we just mark the audit row purged.
      target&.destroy
    rescue => e
      Rails.logger.error("PurgeExpiredDeletionsJob failed for #{deletion.id}: #{e.message}")
    end
  end
end

  end

  member_action :restore, method: :post do
    deletion = Deletion.find(params[:id])
    target = deletion.trashable
    if target.nil?
      redirect_to madmin_deletion_path(deletion), alert: "Record already purged — cannot restore"
    else
      Current.admin = current_admin
      target.restore
      redirect_to madmin_deletion_path(deletion), notice: "Restored"
    end
  end

  member_action :purge_now, method: :post do
    deletion = Deletion.find(params[:id])
    target = deletion.trashable
    deletion.update!(purged_at: Time.current)
    target&.destroy
    redirect_to madmin_deletions_path, notice: "Purged"
  end
end
```

module DeletionHelper
  # Renders a name + deactivated badge if the record is trashed.
  # Use this in any view where the counterparty might see a trashed user.
  def trashable_display(record, size: :default)
    return "" unless record

    name_span = content_tag(:span, record.display_name, class: trashed_text_class(record))
    if record.respond_to?(:trashed?) && record.trashed?
      name_span + render("shared/deactivated_badge", record: record, size: size)
    else
      name_span
    end
  end

  private

Insert after the existing Multilingual section:

```markdown
## Safe Delete (Trashable)

Business-valuable records (User, Team, Membership, Conversation, ConversationMessage, ConversationParticipant, Chat, Message) are soft-deleted via the `Trashable` concern. Ephemeral records (Article, Language, TeamLanguage, Noticed::*, Setting, Admin, ProviderCredential) stay on hard-delete.

### Making a model Trashable

```ruby
class Card < ApplicationRecord
  include Trashable
  trashable_snapshot :title, :author_name
  trashable_cascade :comments
end
```

### Lifecycle

- `card.trash(by: user, reason: "spam", source: "ui")` — creates a Deletion row, freezes the snapshot, cascades to declared associations
- `card.trashed?` — true if a Deletion row exists
- `card.restore` — destroys the Deletion row and restores any cascaded siblings in the same cascade group
- `PurgeExpiredDeletionsJob` runs daily at 3am UTC and hard-deletes records whose `deletion.expire_at` has passed

### Scopes

- `Card.kept` — excludes trashed (use this in user-facing lists)
- `Card.trashed` — only trashed
- `Card.with_trashed` — everything (use in Madmin)

There is NO default scope. Controllers must opt into `.kept` explicitly. This is deliberate: `current_user.card.find(params[:id])` must still work on a trashed card so restore paths can resolve it.

### Retention

Per-model via `Setting.<model>_retention_days`, defaults 30–90 days. Edit in Madmin at `/madmin/settings`.

### Counterparty display

When rendering a user/team name in a view where another user might see it, use the `trashable_display(record)` helper. It returns the frozen snapshot with an "Deactivated" badge when the record is trashed, so counterparty views never 500 or show blank.

### Audit

Every trash creates a `Deletion` row with `deleted_by`, `reason`, `source` (ui/mcp/madmin/cascade/job), frozen `display_snapshot`, and a `cascade_group_id`. The row survives the actual purge — the `trashable` polymorphic association becomes nil, but the audit record stays forever. Queryable at `/madmin/deletions`.

### Rule: don't add a `Deleted*Presenter` class

The `display_name`/`display_email`/`display_avatar_url` helpers on the concern ARE the presenter. Don't build a parallel `DeletedUserPresenter` wrapper.

### Rule: don't `default_scope` trashed rows

We deliberately don't install a default scope. Apps that want list-style hiding call `.kept` explicitly. This lets lookups, admin views, and restore paths find trashed records naturally.
```
